# SES_04_Lab02

## 1 - DAST with OWASP ZAP Automation Framework

**Objective:** Configure and run a Dynamic Application Security Testing (DAST) scan against a deliberately vulnerable application (OWASP Juice Shop) using the OWASP ZAP Automation Framework via Docker.

### Prerequisites

Before starting, ensure you have the following installed on your machine:

* **Docker**
* **Docker Compose**
* **Git** (optional, but recommended for code management)

---

### 1.1 Set Up the Environment

First, let's create a new folder for our DAST project and set up the necessary directories.
Open your terminal and run:

```bash
mkdir dast-lab
cd dast-lab
mkdir zap-reports
```

We create a `zap-reports` directory to store the HTML reports generated by the ZAP scanner.

### 1.2 Define the Vulnerable Target and Scanner

We will use `docker-compose` to spin up both our target application (Juice Shop) and our DAST scanner (ZAP) in the same network so they can communicate.

Create a file named `docker-compose.yaml` in the `dast-lab` folder and paste the following configuration:

```yaml
services:
  # 1. The Target Application (Vulnerable)
  juice-shop:
    image: bkimminich/juice-shop:latest
    container_name: juice-shop
    ports:
      - "3000:3000"
    networks:
      - dast-network
    # Healthcheck to ensure Juice Shop is ready to receive traffic before ZAP starts
    healthcheck:
      test: [ "CMD", "/nodejs/bin/node", "-e", "fetch('http://localhost:3000/').then(r => process.exit(r.ok ? 0 : 1)).catch(() => process.exit(1))" ]
      interval: 10s
      timeout: 5s
      retries: 12

  # 2. The DAST Scanner (OWASP ZAP)
  zap-scanner:
    image: zaproxy/zap-stable:latest
    container_name: zap-scanner
    depends_on:
      juice-shop:
        condition: service_healthy
    # Map the local folder for the container to read the plan and save the report
    volumes:
      - ./zap-reports:/zap/wrk/:rw
    user: "0:0" # Run as root to avoid permission issues when writing the report
    # Run ZAP in 'headless' mode using our YAML plan
    command: zap.sh -cmd -autorun /zap/wrk/zap-plan.yaml
    networks:
      - dast-network

networks:
  dast-network:
    driver: bridge
```

### 1.3 Create the ZAP Automation Plan

OWASP ZAP allows us to script our attacks using YAML files. We need to tell ZAP what URL to attack, how to discover pages (Spider), and what attacks to perform (Active Scan).

Inside the `zap-reports` directory, create a file named `zap-plan.yaml` and paste this configuration:

```yaml
env:
  contexts:
  - name: "juice-shop"
    urls:
    - "http://juice-shop:3000"
    includePaths: []
    excludePaths: []
  parameters:
    failOnError: false
    failOnWarning: false
    progressToStdout: true
jobs:
# Step 1: Discover pages in the application (Spider)
- type: spider
  parameters:
    context: "juice-shop"
    url: "http://juice-shop:3000"
    maxDuration: 1 # Run spider for a maximum of 1 minute

# Step 2: Actively attack the discovered pages (Active Scan)
- type: activeScan
  parameters:
    context: "juice-shop"

# Step 3: Generate a readable HTML report
- type: report
  parameters:
    template: traditional-html
    reportDir: /zap/wrk
    reportFile: zap-report.html
    reportTitle: "ZAP Automation Framework Report"
    reportDescription: "Results from ZAP Autoframework scan of Juice Shop"
```

### 1.4 Execute the Scan

Now that we have our target, scanner, and attack plan configured, it's time to run the assessment!

Start the Juice Shop application in the background:

```bash
docker compose up -d juice-shop
```

Wait a few seconds for it to become healthy, and then start the ZAP scanner:

```bash
docker compose up zap-scanner
```

**What happens?**
Your terminal will show ZAP starting up. It will read the `zap-plan.yaml`, start sending requests to find all the different pages in the Juice Shop (Spider), and then actively inject payloads (like SQL Injection or XSS) into every input field it finds (Active Scan).

You might see errors related to `Error: Only .md and .pdf files are allowed!` in your Juice Shop container logs. This is normal! It means ZAP is successfully testing for Directory Traversal vulnerabilities.

### 1.5 Analyze the Results

Once the `zap-scanner` container finishes running and exits, check your local `zap-reports` directory.

1. You should see a new file called `zap-report.html`.
2. Open this file in your web browser.
3. Review the vulnerabilities discovered by ZAP. You'll likely find several high-severity issues, such as Cross-Site Scripting (XSS), SQL Injection, and more.

**Congratulations!** You just used a DAST tool to automatically discover vulnerabilities in a running application from an attacker's perspective.

## 2 - Personalize workflows in ZAP Automation Framework

**Objective:** Learn how to customize the ZAP automation plan to add rules, limit scan times, or change the reporting format.

### 2.1 Understanding the Automation Framework

The OWASP ZAP Automation Framework (AF) allows you to define a sequence of jobs in a YAML file. These jobs run sequentially, passing data between them if needed.

By default, we used a very basic pipeline in Part 1:

1. `spider`: Crawls the application to find all endpoints.
2. `activeScan`: Attacks the discovered endpoints.
3. `report`: Generates the HTML output.

### 2.2 Adding an Authentication Job

In real-world applications, most vulnerabilities hide behind a login screen. To properly test our application, we must teach ZAP how to authenticate.

OWASP Juice Shop uses **JSON-based authentication** on the `/rest/user/login` endpoint. We need to update our `zap-plan.yaml` to include the authentication method, login credentials, and how ZAP can verify if it's currently logged in.

Update the `contexts` section in your `zap-plan.yaml` inside the `zap-reports` directory to look like this:

```yaml
env:
  contexts:
  - name: "juice-shop"
    urls:
    - "http://juice-shop:3000"
    includePaths: []
    # Tell ZAP to NEVER scan or click on the logout path to avoid losing the session
    excludePaths: 
    - "http://juice-shop:3000/rest/user/logout"
    
    # Define Authentication Strategy (Juice Shop uses JSON authentication)
    authentication:
      method: "json"
      parameters:
        loginPageUrl: "http://juice-shop:3000/#/login"
        loginRequestUri: "http://juice-shop:3000/rest/user/login"
        loginRequestBody: '{"email":"{%username%}","password":"{%password%}"}'
      # How ZAP verifies if the login was successful by looking at the response body
      verification:
        method: "response"
        loggedOutRegex: "401 Unauthorized"
        loggedInRegex: ".*token.*"
        
    # Define the Users
    users:
    - name: "admin-user"
      credentials:
        # Default Juice Shop admin credentials
        username: "admin@juice-sh.op"
        password: "admin123"
```

Once authentication is defined in the `env`, you must also tell your `spider` and `activeScan` jobs to actually *use* this user context by passing the `user: "admin-user"` parameter:

```yaml
- type: spider
  parameters:
    context: "juice-shop"
    user: "admin-user"  # <-- Tell the Spider to run as Admin!
    url: "http://juice-shop:3000"
    maxDuration: 1
```

### 2.3 Fine-tuning the Active Scan

Active scanning can take a long time and might execute attacks you don't care about. We can configure the `activeScan` job to use specific policies or limit its speed. We also need to tell the Active Scanner to use the user we defined above!

Modify the `activeScan` job in your `zap-plan.yaml` to look like this:

```yaml
- type: activeScan
  parameters:
    context: "juice-shop"
    user: "admin-user"  # <-- Tell the Active Scanner to also run as Admin!
    # Don't let the active scan run forever! Limit each rule to 1 minute
    maxRuleDurationInMins: 1
    # Limit the ENTIRE active scan to 2 minutes for classroom efficiency
    maxScanDurationInMins: 2
  policyDefinition:
    # Set the overall intensity of the scan (Low, Medium, High, Insane)
    defaultStrength: Low
    # Set how easily ZAP will report a vulnerability (Low, Medium, High)
    defaultThreshold: Medium
```

*(By setting the strength to `Low`, ZAP sends fewer payloads per parameter, making the scan finish much faster, though it might miss some edge-case vulnerabilities.)*

### 2.4 Generating Multiple Reports

Sometimes developers want an HTML report for their eyes, but the CI/CD pipeline needs a JSON report to parse programmatically. ZAP can generate both simultaneously!

Modify the `report` job at the bottom of your `zap-plan.yaml` to generate two different reports:

```yaml
- type: report
  parameters:
    template: traditional-html
    reportDir: /zap/wrk
    reportFile: zap-report.html
    reportTitle: "ZAP Automation Framework Report"
- type: report
  parameters:
    template: risk-confidence-html
    reportDir: /zap/wrk
    reportFile: risk-report.html
    reportTitle: "ZAP Risk/Confidence Report"
```

*(You can see all available templates by running ZAP Desktop, but the standard `traditional-html`, `risk-confidence-html`, `traditional-xml`, and `traditional-json` are always available).*

### 2.5 Run the Customized Scan

After editing and saving your `zap-plan.yaml`, simply re-run the ZAP container:

```bash
docker compose up zap-scanner
```

Once it finishes, check your `zap-reports` folder. You should now see two different HTML reports! Compare `zap-report.html` and `risk-report.html` to see how the same data can be presented in different formats for different audiences.

## 3 - DAST with CI/CD (GitHub Actions)

**Objective:** Integrate the OWASP ZAP DAST scanner into a continuous integration (CI) pipeline using GitHub Actions. Instead of manually running tests on our local machine, we will configure GitHub to automatically spin up our vulnerable application and scan it every time code is pushed.

### 3.1 Push the Project to GitHub

If you haven't already, ensure your `dast-lab` folder (containing the `docker-compose.yaml` and the `zap-reports` folder with the `zap-plan.yaml`) is pushed to a repository on GitHub.

### 3.2 Create the GitHub Actions Workflow

In your local repository, create the standard directory structure for GitHub Actions:

```bash
mkdir -p .github/workflows
```

Now, create a new file named `.github/workflows/dast-pipeline.yml` and paste the following configuration:

```yaml
name: Continuous DAST Pipeline (OWASP ZAP)

# Trigger the workflow on pushes to the main branch or manual dispatch
on:
  push:
    branches: [ "main" ]
  workflow_dispatch:

jobs:
  security-scan:
    name: ZAP DAST Scan
    runs-on: ubuntu-latest
    
    steps:
      # 1. Download the code from the repository
      - name: Checkout Code
        uses: actions/checkout@v4

      # 2. Start the Target Application (Juice Shop)
      # We run this in detached mode (-d) so the pipeline can continue to the next step
      - name: Start Juice Shop Container
        run: |
          docker compose up -d juice-shop
          
      # 3. Wait for the Application to be Healthy
      # Since Juice Shop can take a few seconds to start, we must wait before attacking it
      - name: Wait for Target Application
        run: |
          echo "Waiting for Juice Shop to become healthy..."
          while ! curl -s http://localhost:3000 > /dev/null; do
            echo "Waiting..."
            sleep 5
          done
          echo "Juice Shop is UP!"

      # 4. Run the ZAP Scanner using docker compose
      # This reuses the exact same zap-plan.yaml you created in Exercises 1 and 2!
      - name: Run ZAP DAST Scan
        run: |
          docker compose up zap-scanner
          
      # 5. Save the Vulnerability Report
      # GitHub Actions destroys the virtual machine when finished. 
      # We must upload the generated HTML report as an "Artifact" so we can download it later.
      - name: Upload ZAP Report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: dast-zap-report
          path: zap-reports/zap-report.html
          retention-days: 7
```

### 3.3 Commit and trigger the Pipeline

Save the `.github/workflows/dast-pipeline.yml` file, commit it, and push it to your `main` branch:

```bash
git add .github/workflows/dast-pipeline.yml
git commit -m "Add continuous DAST pipeline"
git push
```

### 3.4 Analyze the CI/CD Results

1. Go to your repository on GitHub.
2. Click on the **Actions** tab at the top.
3. You should see your `Continuous DAST Pipeline (OWASP ZAP)` running or already completed.
4. Click on the workflow run precisely. You will see all the steps executing: starting the Juice Shop container, waiting for it to be healthy, and finally running ZAP.
5. At the bottom of the summary page, under the **Artifacts** section, you will find `dast-zap-report` ready to be downloaded!

**The Real-World Impact:**
By inserting this pipeline into a repository, Security Teams ensure that every new feature or Pull Request proposed by developers is automatically dynamically tested before ever reaching Production. If the ZAP Action is configured with `fail_action: true`, it physically prevents merging vulnerable code!
